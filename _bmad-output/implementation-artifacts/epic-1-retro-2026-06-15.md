## What Went Well

**Amelia (Developer):** The monorepo scaffold (1.1) and OpenAPI envelope (1.2) paid off immediately — every later story plugged into the same `{ data, meta }` contract and `packages/api-client` regeneration workflow.

**Winston (Architect):** Standardising on **`IConnectionMultiplexer` for Redis auth state** (OTP, refresh, password reset) after the 1.4 `IDistributedCache` prefix lesson eliminated an entire class of production bugs. Lua atomic consume scripts in 1.5 and 1.9 are a repeatable pattern for Epic 2+.

**John (PM):** FR-1 and FR-2 are genuinely complete: OTP login, refresh rotation, session kill, RBAC deny paths, and self-service reset with **no account enumeration** on forgot-password.

**Sally (UX):** Web and mobile auth flows consistently use **`aria-live` / `accessibilityLiveRegion`** and EXPERIENCE.md copy. Role routing (supervisor web vs mobile-only field roles) landed cleanly in 1.6.

**Dana (QA):** `AuthWebApplicationFactory` + Testcontainers + `FakeEmailSender` gave us **real integration tests** without SMTP. Adversarial code review after each story caught HIGH issues before they compounded (1.7 bearer token, 1.8 JWT role claim, 1.9 enumeration leak).

---

## Challenges & Growth Areas

**Charlie (Senior Dev):** Early stories underestimated Redis complexity. 1.4 deferred refresh hardening; 1.5 paid the interest with Lua scripts, peek-then-consume, and reuse detection.

**Elena (Junior Dev):** Mobile auth had subtle bugs — stale Bearer on public paths, OTP challenge lost on cold start, session-expired not firing after retried 401. Each required targeted tests, not just happy-path coverage.

**Alice (Product Owner):** Security requirements (enumeration-safe responses, deactivated-user silence on forgot-password) are easy to violate accidentally. We caught two enumeration risks in review (login timing deferred; forgot-password 500 in 1.9).

**Amelia (Developer):** Story specs improved over the epic (validation reports, mandatory reset order, `guestGuard` exclusions), but **brownfield tables in each story** were essential — without them, agents re-read the whole codebase every time.

## Recurring Review Themes (across stories)

| Theme | Stories | Lesson |
|-------|---------|--------|
| Redis storage via multiplexer, not distributed cache | 1.4, 1.5, 1.9 | Always use `IConnectionMultiplexer` for auth tokens |
| Atomic token consume (Lua) | 1.5, 1.9 | Peek-then-act patterns need atomic delete + revoked marker |
| Mobile `PUBLIC_AUTH_PATHS` | 1.7, 1.9 | Extend on every new anonymous auth endpoint |
| Enumeration-safe API responses | 1.4, 1.9 | Generic 200/401 shapes; never 503 on forgot-password |
| Integration test isolation | 1.5, 1.8 | Reset test users (`TokenVersion`, `IsActive`, role) between runs |
| OpenAPI + api-client regen | 1.2–1.9 | `EXPORT_OPENAPI_PATH` + `API_OPENAPI_FILE` after API changes |
| RBAC + deactivated user order | 1.8 | `ActiveUserAuthorizationHandler` before role checks; explicit `RoleClaimType` |

## Team Agreements

- **Redis auth state:** always `IConnectionMultiplexer`, never `IDistributedCache` for tokens
- **Security defaults:** enumeration-safe responses; deactivated users get generic success on recovery flows
- **Mobile:** every new anonymous auth route goes in `PUBLIC_AUTH_PATHS`
- **Web:** recovery routes (`/reset-password`) never use `guestGuard`
- **Tests:** document baseline counts in story DoD; never merge with failing suites
- **Review before done:** no story moves to `done` without code review pass

## Key Takeaways

1. **Invest early in integration test infrastructure** — Testcontainers + fake email paid dividends across 9 stories.
2. **Adversarial review catches security regressions** — enumeration leaks and JWT claim mapping would have shipped without review.
3. **Repeatable Redis token patterns** — Issue/peek/consume/revoked marker is now a template for any single-use token in the platform.
4. **Story brownfield context reduces rework** — validation reports and "do not break" baselines kept parallel web/mobile/API work aligned.
