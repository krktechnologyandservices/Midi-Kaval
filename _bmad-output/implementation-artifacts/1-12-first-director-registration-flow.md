# Story 1.12: First Director Registration Flow

Status: done

## Story

As a recipient of an activation link,
I want to open the link, set my name and password, and activate as the first Director of my organisation,
So that I can begin using Midi-Kaval as the administrator.

## Acceptance Criteria

1. **Given** the activation link was generated by the Vendor (Story 1.11)
   **When** the recipient opens the link (`/activate?token={raw}&sig={hmac}`)
   **Then** an unauthenticated registration page is displayed
   **And** the recipient's email is pre-filled (extracted from the activation token's `target_email`)
   **And** the HMAC signature is verified first — if invalid (tampered), return 400: "Invalid activation link."

2. **Given** a valid activation link is opened
   **When** the recipient sets their full name and password
   **Then** the password must meet minimum policy: 8+ characters, 1 uppercase letter, 1 lowercase letter, 1 digit
   **And** password complexity requirements are displayed before submission
   **And** an error is shown if password does not meet policy
   **And** the full name is required (1-256 characters)

3. **Given** the registration form is submitted with valid inputs
   **When** the activation token is verified server-side
   **Then** the system performs these checks in order:
   - **HMAC signature verification** — reject with 400 if tampered
   - **Look up SHA-256 token hash** in `activation_tokens` table — reject with 400 if not found (or already consumed/expired and cleaned)
   - **Check `expires_at_utc`** — if expired, return 400: "This link has expired or already been used. Please contact the Vendor to request a new activation link."
   - **Check `consumed_at_utc`** — if already consumed, return 400 with the same message as expired (do not reveal which one)
   - **Mark `consumed_at_utc = NOW()`** — single-use enforced atomically
   - Create a new `User` row with `role = Director`, `organisation_id` set to the token's `OrganisationId`
   - Set the `Organisation.is_active = true` on the organisation linked to the token
   - All persistence happens in a single DB transaction — if any step fails, everything is rolled back (no orphaned tokens or partial registrations)

4. **Given** the registration succeeds
   **When** the response is returned
   **Then** the organisation is set to `is_active = true`
   **And** the user account exists with `role = Director`
   **And** the user can log in immediately with their email and chosen password
   **And** the API returns 200 with a success message: "Your organisation is active. Welcome to Kaval."
   **And** the frontend displays a success page with a "Go to Login" button

5. **Given** the activation link is expired or already consumed
   **When** the recipient attempts registration
   **Then** a clear error message is shown: "This link has expired or already been used. Please contact the Vendor to request a new activation link."
   **And** the page does not reveal whether the link was expired or already consumed (prevent enumeration)

6. **Given** the activation endpoint is called
   **When** requests exceed the rate limit
   **Then** returns HTTP 429 with `Retry-After` header
   **And** the rate limit is 10 req/min per IP (matching the pattern from Story 1.11)

- [x] **Create RegistrationService** — `apps/api/Domain/RoleManagement/RegistrationService.cs`

### Token consumption flow (critical)

The `TokenService.ValidateSignature()` method (created in Story 1.11) is shared for HMAC verification. The full consumption sequence is:

1. **HMAC verify**: `TokenService.ValidateSignature(rawToken, signature)` — if false, reject with 400 (tampered)
2. **SHA-256 hash**: `SHA256.HashData(Encoding.UTF8.GetBytes(rawToken))` → hex-encoded → lookup `ActivationToken` by `TokenHash` in a single query with `.Include(t => t.Organisation)`
3. **Expiry check**: `token.ExpiresAtUtc <= DateTime.UtcNow` → if expired, reject
4. **Consumed check**: `token.ConsumedAtUtc != null` → if consumed, reject (same message as expired)
5. **Atomic consume**: Use raw SQL `UPDATE activation_tokens SET consumed_at_utc = NOW() WHERE id = @id AND consumed_at_utc IS NULL` via `await db.Database.ExecuteSqlInterpolatedAsync(...)`. Check the return value — if 0 rows affected, another request consumed it first → throw concurrency error. This avoids needing a `RowVersion` concurrency token.
6. **Create user**: `User` with `Role = Director`, `OrganisationId = token.OrganisationId`
7. **Activate org**: `organisation.IsActive = true`
8. **Commit**: Single `SaveChangesAsync` — if any step fails after the update, everything rolls back

### Password validation utility

Implement password validation as a static helper or inline in `RegistrationService`:

```csharp
public static (bool IsValid, string[] Errors) ValidatePassword(string password)
{
    var errors = new List<string>();
    if (password.Length < 8)
        errors.Add("Password must be at least 8 characters long.");
    if (!password.Any(char.IsUpper))
        errors.Add("Password must contain at least one uppercase letter.");
    if (!password.Any(char.IsLower))
        errors.Add("Password must contain at least one lowercase letter.");
    if (!password.Any(char.IsDigit))
        errors.Add("Password must contain at least one digit.");
    return (errors.Count == 0, errors.ToArray());
}
```

The same validation must be applied client-side (Angular Reactive Forms validators) AND server-side (in the domain service).

### Activation link format

### Orphaned / stale tokens

- Tokens that are neither consumed nor expired remain in the database — they become inert after their expiry date and cannot be used
- A future cleanup job (Story 1.13+) can remove expired tokens
- The same "expired or already used" message prevents token enumeration attacks

### Concurrent consumption protection

Two HTTP requests arriving simultaneously with the same valid token must not both succeed. Since `ActivationToken` does not have a `RowVersion` concurrency token, EF Core's default optimistic concurrency cannot detect the race. Use raw SQL for the consume step:

```csharp
// After all validation checks pass (HMAC, expiry, consumed_at from read)
var rows = await db.Database.ExecuteSqlInterpolatedAsync(
    $"UPDATE activation_tokens SET consumed_at_utc = NOW() WHERE id = {tokenId} AND consumed_at_utc IS NULL",
    cancellationToken);

if (rows == 0)
{
    // Another request consumed this token first
    throw new InvalidOperationException("This link has already been used.");
}
```

This atomic `UPDATE ... WHERE consumed_at_utc IS NULL` ensures that only the first request wins. The subsequent entity creation and org activation happen inside the same `SaveChangesAsync` transaction (begun via `BeginTransactionAsync()`). If any post-consume step fails, the entire transaction rolls back — but the token remains consumed (the raw SQL UPDATE is inside the transaction).

#### decision-needed

- [x] [Review][Decision] **ForwardedHeaders trusts all proxies** — `ForwardedHeadersOptions` has no `KnownProxies`/`KnownNetworks` configured, so any client can spoof `X-Forwarded-For`. This bypasses IP-based rate limiting on unauthenticated endpoints. Requires proxy infrastructure knowledge to configure correctly. [`apps/api/Program.cs:112-115`]

- [x] [Review][Decision] **Token/Sig exposed in URL query params on GET endpoint** — `GET /api/v1/auth/activate?token=...&sig=...` leaks secrets into server logs, browser history, and Referer headers. The activation token grants organisation activation rights. Options: (a) accept the URL pattern as required by the email link flow but remove the GET validation endpoint (validate only client-side), (b) use a hash-based approach, or (c) keep as-is with mitigation notes. [`apps/api/Controllers/V1/Auth/RegistrationController.cs:23-24`]
- *Resolution: Accepted as-is — the URL pattern is inherent to the email link flow; the GET endpoint is read-only validation while the mutation happens via POST body.*
- [x] [Review][Decision] **Two-step flow merged into single view** — The spec describes separate Step 1 (verify email) → Step 2 (set name/password) with a step boundary. The implementation shows email as read-only alongside the form on one card. Options: (a) keep merged as simpler UX (email is low-interaction), (b) implement true two-step with "Continue" button. [`apps/web/src/app/features/activation/activation.component.html`]
- *Resolution: Implemented true two-step — validation resolves to an info step with a "Continue to setup" button before the form appears.*
