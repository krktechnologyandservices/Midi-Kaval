FAIR

This PRD is a well-structured, decision-rich capability spec. Its strengths are decision-readiness, scope honesty, and downstream usability — every FR has testable consequences, the Glossary is consistent, and the decision log anchors the key trade-offs. However, the adversarial review surfaced critical gaps in data model, session management, and 2FA recovery that make this a good conversation starter rather than a buildable specification. The PRD systematically specifies UI/frontend behavior while hand-waving the backend mechanisms that make it work.

Adversarial review

review-adversarial-general.md

This PRD is a good conversation starter but not a buildable specification. It correctly identifies a real pain point and proposes a sensible high-level architecture. However, the document has a systematic pattern of specifying UI/frontend behavior while hand-waving the backend mechanisms that make it work. The five critical findings share a common root cause: the PRD describes what should happen without specifying how it must work at the data and infrastructure level.

critical

No Director 2FA recovery flow

FR-10, OQ #4, §7.2
2FA is mandated but SMS/email backup codes deferred to v2, and recovery process is an open question. Every 2FA loss becomes a support escalation, contradicting the PRD's goal of reducing vendor dependency.
Fix: Define the recovery path — even if it's "Director contacts Vendor support, Vendor verifies identity and resets 2FA enrollment." Document the choice.
critical

Permanent deletion semantics are contradictory

FR-8, FR-13, AS-3
FR-8 says "removed or anonymised", FR-13 requires identifying deleted users in audit events, AS-3 says anonymises. These are mutually exclusive and determine the entire data model.
Fix: Pick one path (recommended: anonymise with identity snapshot in audit event) and make it consistent across FR-8, FR-13, AS-3, and the Glossary.
critical

Session termination for suspension has no feasible mechanism

FR-6, AS-2
5-minute SLA on terminating sessions is specified without defining the session management strategy. With JWT tokens (common pattern), there's no server-side session to revoke.
Fix: Specify the session management approach (short-lived JWT with blacklist, server-side sessions, or adjust the SLA). Document architectural implications.
critical

Vendor Backstage is a second undefined application

FR-1, FR-3, §2.1, OQ #2
Treated as a minor requirement but requires its own auth, user management, UI, authorization model, and deployment — roughly half the Director dashboard scope, occupying ~10% of the PRD.
Fix: Scope the Vendor Backstage properly — define its own FRs for authentication, user management, and UI requirements, or explicitly reduce its scope to a simple CLI/admin script.
critical

No data model defined

entire document
No entity-relationship definitions, schema, or mapping to the existing Midi-Kaval data model. Every team member will make different assumptions about how organisations, users, roles, tokens, and audit events relate.
Fix: Add a data model section with entity definitions, key relationships, and how they map to the existing database schema.
high

Rate limiting underspecified

§5 Security NFR
Single sentence with no thresholds, burst handling, endpoint differentiation, or HTTP response code.
high

Email delivery failure handling disconnected from FRs

§5, FR-1, FR-3, FR-5, FR-11, FR-12, FR-15, FR-16
Every FR assumes email delivery always works. No retry strategy per email type, no delivery status visibility, no bounce handling.
high

Password complexity is a dangling dependency

FR-2, OQ #1
FR-2 requires "organisation-configured" policy but no configuration mechanism exists. Limbo between system-wide default and per-org config.
high

No existing-user migration or cutover strategy

entire document
Hardcoded accounts acknowledged in Vision but no migration path, coexistence plan, or rollback described.
high

Audit broadcast creates notification spam

FR-15
N Directors × M actions = N*M emails with no digest, no rate limiting, no opt-out. Notification fatigue guaranteed.
high

"Immutable audit log" not defined

FR-13, Glossary
Implementation spectrum from plain DB table to cryptographic chain — radically different security, performance, and query properties.
high

1-hour Safety Net detection too slow

FR-3, AS-1
Last-Director loss means unmanaged org for up to 1 hour. Event-driven detection not considered.
high

SM-2 is poorly defined

§8, SM-2
"Zero support tickets" conflates product quality, documentation, user patience, and ticket classification. Incentivizes wrong behavior.
